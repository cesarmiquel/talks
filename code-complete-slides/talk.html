Razones válidas para crear una rutina

Una lista breve

Algunas de las razones más importantes/comunes para armar una rutina:

  • Reducir la complejidad. Permite no tener que recordar los detalles de como funciona.
  • Introducir una abstracción intermedia/entendible
  • Evitar código duplicado
  • Esconder secuencias
  • Mejorar la portabilidad
  • Simplificar lógica booleana complicada / reglas de negocios
  • Mejorar el rendimiento

Razones válidas para crear una rutina:

Introducir una abstracción intermedia

        if ( node <> NULL ) then
          while ( node.next <> NULL ) do
            node = node.next
            leafName = node.name
          end while
        else
          leafName = ""
        end if

podría reemplazarse por:

      leafName = GetLeafName( node )

Razones válidas para crear una rutina:

Operaciones demasiado simples

      points = deviceUnits * ( POINTS_PER_INCH / DeviceUnitsPerInch() )

podría reemplazarse por:

      points = DeviceUnitsToPoints( deviceUnits )

Razones válidas para crear una rutina:

Operaciones demasiado simples

    Function DeviceUnitsToPoints( deviceUnits: Integer ) Integer;
      if ( DeviceUnitsPerInch() <> 0 )
        DeviceUnitsToPoints = deviceUnits *
          ( POINTS_PER_INCH / DeviceUnitsPerInch() )
      else
        DeviceUnitsToPoints = 0
      end if
    End Function

Cómo ponerle nombre a nuestras rutinas

Describir todo lo que hace

  • Describir toda la salida y efectos secundarios
  • Ejemplo: si una rutina calcula los totales de un reporte y abre un archivo:

      ComputeReportTotalsAndOpenOutputFile()

    y no solo

      ComputeReportTotals().
  • Si los nombres resultantes son horribles => el problema es la rutina
  • La solución es re-pensar y/o separar la rutina

Cómo ponerle nombre a nuestras rutinas

Evitar nombres que son genéricos o poco específicos

  • Un nombre débil es el síntoma de una rutina con propósito poco claro.
  • Evitar nombres como

      HandleCalculation()
      PerformServices()
      OutputUser()
      ProcessInput()
  • Hay veces que el único problema es la elección del nombre:

      HandleOutput()

    no es bueno pero

      FormatAndPrintOutput()

    está perfecto: describe bien lo que hace la rutina.

Cómo ponerle nombre a nuestras rutinas

Para los procedimientos: usar un verbo seguido por un sujeto

  • Buenos ejemplos:

      PrintDocument()
      CalcMonthlyRevenues()
      CheckOrderlnfo()
      RepaginateDocument()
  • En lenguajes orientados a objetos utilizar verbos ya que el objeto normalmente aparece:

      document.print()
      orderInfo.check()
      monthlyRevenues.calc()
  •   document.printDocument()es redunante

Cómo ponerle nombre a nuestras rutinas

Usar los opuestos correctamente

  • Utilizar bien los opuestos ayuda a la consistencia
  • La consistencia ayuda a la legibilidad
  • Opuestos como first/last son claros
  • Funciones como:

     FileOpen()

    y

     _lclose()

    son asimétricos y generan confusión

Rutinas de alta calidad

Largo óptimo de una rutina

  • El libro presenta varios estudios que estudiaron correlaciones entre largo y métricas de calidad
  • En la mayoría se obtienen mejores métricas con rutinas cortas
  • El óptimo es 50-150 lineas
  • nunca deberían exeder las 200 lineas

Rutinas de alta calidad

Recomendaciones para los parámetros de las rutinas

  • Poner los parámetros en órden: entrada, entrada-salida y sólo salida
  • Si hay varias rutinas con los mismos parámetros: usar el mismo orden
  • No usar más de 7 parámetros
  • Utilizar todos los parámetros
  • Documentar los supuestos que asumimos de los parámetros
  • Mejor aún: utilizar asserts()
  • Qué cosas documentar?
    • Tipo de parámetros
    • Unidades (metros, pulgadas, segundos, etc)
    • Rangos esperados
    • Valores que no deberían aparecer

Variables

Recomendaciones

  • Declarar todas las variables
  • Usar convenciones para prefijos/sufijos (i.e. NoCards o NumCards)
  • Si hay utilizar herramientas para detectar violaciones a las convenciones
  • Initicializar cada variable lo más cerca posible del primer uso.
  • Usar todas las variables que se definan

Variables

Inicializar lo más cerca del 1er uso

        // initialize all variables
        var accountIndex = 0
        var total = 0.0
        var done = false
        
        // ...
        
        // code using accountIndex
        // ..
        
        // code using total
        // ...
        
        //  code using done
        while( ! done ) {
        
        }

Variables

Inicializar lo más cerca del 1er uso

        var accountIndex = 0
        // code using accountIndex
        // ..
        
        var total = 0.0
        // code using total
        // ...
        
        var done = false
        //  code using done
        while( ! done ) {
        
        }
  • En el 1er caso done el código anterior lo pudo modificar accidentalmente
  • Juntar las inicializaciones da la impresión de que todas las variables se usan en todo el código.
  • Si después se agregan loops alrededor del loop de done se pueden olvidar de re-inicializarla

Variables

Más tips

  • Usar _final_ o _const_ cuando esta disponible. Util para constantes de clases, parámetros que no se modifican, cualquier variable que no debería ser modificada por la rutina.
  • Prestar particlar atención a los contadores: un error común es olvidar resetearlos cuando se reutilizan
  • Aprovechar los warnings de los compiladores
  • Minimizar el scope de las variables. Por la misma razón por las que no se usan las globales
  • Cuánto más grande el es scope de una variables más dificil de entender

Variables

Variables 'temporales'

Cada variable deber tener un sólo uso.


        // Compute roots of a quadratic equation.
        // This code assumes that (b*b-4*a*c) is positive.
        temp = math.sqrt( b*b - 4*a*c )
        root[0] = ( -b + temp ) / ( 2 * a )
        root[1] = ( -b - temp ) / ( 2 * a )
        ...
        // swap the roots
        temp = root[0]
        root[0] = root[1]
        root[1] = temp
  • Cual es la relación entre temp en el primer caso con el segundo? NADA.
  • Entonces: por qué se llaman igual!?
  • Usar el mismo nombre da la impresión de que están relacionadas

Variables

Variables 'temporales' (cont.)

Simplemente cambiando el nombre de las variables es mucho más evidante qué hace cada variable.


        // Compute roots of a quadratic equation.
        // This code assumes that (b*b-4*a*c) is positive.
        discriminant = b*b - 4*a*c
        root[0] = ( -b + math.sqrt( discriminant ) ) / ( 2 * a )
        root[1] = ( -b - math.sqrt( discriminant ) ) / ( 2 * a )
        ...
        // swap the roots
        oldRoot = root[0]
        root[0] = root[1]
        root[1] = oldRoot

Variables

Variables con significados ocultos

Evitar variables con significados ocultos. Por ejemplo:

  • pageCount es el número de páginas impresas pero si es -1 indica un error
  • customerId es el ID de cliente pero si es mayor que 500.000, restando 500.000 obtenes el ID de la cuenta!
  • bytesWritten es la cantidad de bytes escritos pero si es negativo es el volumen donde se escriben los datos
  • En el 1er caso el -1 está haciendo de booleano
  • Aunque para uno puede resultar evidente el doble uso para otra persona seguramente le resultará confuso. Usar dos variables.

Variables

Bautizando variables

  • El nombre debe represantar de manera precisa la entidad que va a representar
  • Una buena técnica es decir en voz alta los que representa. Ese es el nombre
  • Running total of checks written to date: runningTotal, checkTotal en lugar de ct checks CHKTTL c1
  • Velocity of a bullet train: velocity, trainVelocity, velocityInMph en lugar de velt, v, tv, x, x1, x2, train
  • Current date: currentDate, todaysDate en lugar de cd, current, c, x, x1, x2, date
  • Lines per page: linesPerPage en lugar de lpp, lines, l, x, x1, x2

Nombres para variables en loops

  • Cuando el loop es bien corto se pueden usar variables de loops: i, j, k
  • Para loops anidados es mejor usar nombres más largos
    for ( teamIndex = 0; teamIndex < teamCount; teamIndex++ ) {
    for ( eventIndex = 0; eventIndex < eventCount[ teamIndex ]; eventIndex++ ) {
        score[ teamIndex ][ eventIndex ] = 0;
    }
}
  • Es infinito más claro
      score[ teamIndex ][ eventIndex ]
    que
      score[ i ][ j ]

Variables

'status variables'

  • Pensar mejores nombres que flag
  • Horror!

              if ( flag ) ...
          if ( statusFlag & 0x0F ) ...
          if ( printFlag == 16 ) ...

Variables

'status variables'

Mucho mejor:


        if ( dataReady ) ...
        if ( characterType & PRINTABLE_CHAR ) ...
        if ( reportType == ReportType_Annual ) ...
        if ( recalcNeeded == false ) ...

        dataReady = true
        characterType = CONTROL_CHARACTER
        reportType = ReportType_Annual
        recalcNeeded = false

Variables

Recomendaciones para variables Booleanas

Buenos nombres: done, error, found, success o ok
Algunos consejos:

  • Usar nombres que impliquen 'true' o 'false'.
  • Un mal ejemplo: status o sourceFile
  • Qué quiere decir que status sea true?! Que sea true significa que está OK?
  • Otra forma de ver que está bien es cuando son is_*: is_done, is_error o is_found tien sentido.
  • is_status qué es!?!?!?
  • Evitar usar variables 'negativa':
      notFound, notDone, etc.
    Generan este tipo de cosas:
       if ( not notFound ) ...

Variables

Variables booleanas intermedias

Hay veces que es útil generar variables booleanas intermeaidas para simplificar expresiones complicadas dentro de un if.

Ejemplo:

      if ( ( ( document.atEndOfDocument() ) && ( ! inputError ) ) &&
        ( ( MIN_LINES <= lineCount ) && ( lineCount <= MAX_LINES ) ) && 
        ( ! this.errorProcessing() ) ) {

        // do something or other
        // ..

      }

Variables

Variables booleanas intermedias (cont.)

Generando un par de variables booleanas intermedias se logra un código mucho más legible

      var allDataRead = ( document.atEndOfDocument() ) && ( ! inputError );
      var legalLineCount = ( MIN_LINES <= lineCount ) && ( lineCount <= MAX_LINES )

      if ( allDataRead && legalLineCount && ( ! ErrorProcessing() ) ) {

        // do something or other
        // ..

      }

Variables

Seguir convenciones cuando existan

Por ejemplo en Java existen las siguientes convenciones:

  • i y j son indices enteros
  • Las constantes van ALL_CAPS separadas por '_': SOY_UNA_CONSTANTE
  • Class e interface capitalizadas con la primer letra de cada palabra
  • Variables y nombres de métodos en 'camelcase': variableOrRoutineName
  • get y set se usan como prefijo para los getters/setters: getColor() o setTerminal()

Variables

Nombres cortos

Tips para abreviar nombres de variables:

  • Ser consistente y siempre usar el mismo método
  • Usar abreviaciones estandares (diccionario)
  • Sacar todas las vocales (excepto cuando empiece con vocal). Ej: computer -> cmptr, screen -> scrn, integer -> intgr
  • Usar las primeras letras de la palabra
  • No abreviar sacando una letra de la palabra
  • Crear nombres que sean pronunciables
  • Muchos más tips en el libro

Variables

Nombres a evitar

  • Evitar nombres que sean parecidos a otra variable: input e inputValue, recordNum y numRecords, etc.
  • Evitar nombres con distinto sentido per un nombre muy parecido: clientRecs y clientReps
  • Evitar números en las variables (ej: total1 y total2). Lo más probable que total debiera ser un array.
  • Nombres con errores de ortografía: hilite en lugar de hightlight
  • Palabras que comunmente la gente escribe mal: accumulate, defferred, etc.
  • Letras que se parecen mucho (1 y l), (1 e I), (0 y O), etc. Usar un buen font como este: Hermit
    • eyeChartl - eyeChartI
    • ttl5 - ttlS

Aprendiendo por el opuesto

Como escribir código inmantenible

Unmaintainable code está lleno de buenos consejos de lo que NO hay que hacer

Por ejemplo, en la sección de variables

  • Usar el libro '20001 nombres para su bebé' para nombrar las variables
  • Usar siempre (que sea posible) las varibles a, b y c en sus rutinas
  • Si tiene que usar nombres verdaderos cometa pequeñas faltas de ortografía. Pero no siempre.
  • Sea abstracto en el nombre. Ej: routineX48, PerformDataFunction, DoIt, HandleStuff
  • Use _ y __ como nombres de variables
  • Use nombres en otros idiomas, i.e. punkt (Alemán) en lugar del mucho más simple point

Aprendiendo por el opuesto

Como escribir código inmantenible

Mas ejemplos, esta vez de la seccion de comentarios

  • Mienta en los comentarios
  • Documente lo obvio: i = i + 1; // add 1 to i
  • Nunca documente las unidades de medida de variables. Especialmente en programas de ingeniería
  • No borres código obsoleto! Dejalo comentado.
